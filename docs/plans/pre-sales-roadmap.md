# WindowSnap 有料配布に向けた機能洗い出し

## Context（背景・目的）

WindowSnap は、ウィンドウを画面端へドラッグするとスナップする macOS 常駐ユーティリティ。Swift Package Manager 製・630行・外部依存ゼロで、コードの完成度は高いが、**現状は「開発者が自分で使う最小実装」の段階**にとどまる：

- 配布は `git clone` → 自己署名証明書作成 → `swift build` → launchd 登録という開発者向け手順前提
- 設定 UI・ホットキー・有効化トグルなし（カスタマイズはソース定数の書き換え＋再ビルド）
- ライセンス/決済/自動更新/公証など、有料配布の土台が未整備

本プランは、これを**独自サイト直販・グローバル市場・買い切り（将来の有料アップグレード前提）**で売れる商品にするために必要な対応を、2観点で洗い出し優先順位を付けるもの。

### 前提（確認済み）
- 配布チャネル: **独自サイト直販**（Developer ID 署名 + 公証 + ライセンス自前実装が必要）
- 価格モデル: **買い切り・低価格**（BTT より安く。BTT は標準ライセンス ~$22／買い切り ~$45 なので **$3〜8 程度**を想定。+ 将来の有料メジャーアップグレード余地）
- 試用導線: **期間限定トライアル（14日フル機能 → 購入）**。フリーミアムの機能分割はしない
- 操作: **ドラッグ主軸 + 主要スナップの最小ホットキー**（複雑な再割り当て UI は出さない）
- 市場: **グローバル**（英語 UI 基本 + 多言語対応の余地）

### マーケティング方針（最重要・全判断の上位制約）
ターゲットは「**BetterTouchTool は機能が多すぎて使いこなせない／欲しいのはウィンドウスナップだけ／マニアックなカスタマイズは不要**」という層。
- **商品価値は「機能の多さ」ではなく「削ぎ落とし・ゼロ設定で動く・安い」**。機能を足すほど商品コンセプトが薄まる構造。
- よって **"作らない判断" 自体が設計**。下記「スコープ外」を明示的に守る。
- 低価格＝薄利のため、**サポート/インフラのコストを徹底的に低く保つ**（オフライン・ライセンス検証、優れたデフォルト、充実 FAQ で問い合わせを発生させない）。

### 競合の参照線とリスク
Magnet（買い切り ~$5・MAS）、Rectangle Pro（買い切り・直販＋Sparkle）、BetterSnapTool、Swish。直販＋買い切り＋Sparkle は Rectangle Pro と同路線。
- **最大のリスクは BTT ではなく無料の Rectangle / Loop**。「安くシンプル」だけでは無料勢に対して弱い。→ 勝ち筋は **(a) ドラッグ体験の質（プレビュー・吸い付き・誤爆のなさ）の作り込み**、**(b) 設定不要で即動く導入体験**、**(c) BTT 難民への明確なメッセージング**。価格は補助。

### スコープ外（意図的に作らない — 方針からの帰結）
以下は「マニアックなカスタマイズ」に該当し、ターゲットが求めておらず、シンプルさ訴求を毀損するため**実装しない**：
- カスタムグリッド／任意分割レイアウトエディタ
- ウィンドウレイアウトのスナップショット保存・復元（Moom 的機能）
- モニタープロファイル切替
- ホットキーの自由再割り当てマトリクス UI
- 全閾値（EDGE/CORNER/MOVE/TITLEBAR）の GUI 露出（優れたデフォルトで隠蔽）

## 観点1: プロダクトとして必要な機能

「シンプル・ゼロ設定・安い」という商品価値を**損なわずに**有料製品として通用させるために積み増す機能。**機能数は最小限に保ち、体験の質に投資する**のが方針。優先度 P0=これが無いと有料で売れない、P1=体験の質・競争力、P2=余裕があれば。

> 方針からの帰結: 設定 UI は「全閾値の露出」ではなく **優れたデフォルト＋最小トグル**。ホットキーは **主要スナップの固定割り当て**（再割り当てマトリクス UI は出さない）。

### P0（必須 — これが無いと有料製品として成立しない）
1. **最小限の設定 UI（Preferences ウィンドウ）** — `status_item.swift` のメニューに「設定…」を追加。露出する項目は **ライセンスキー入力 / 有効・無効トグル / ログイン時に起動 / ウィンドウ間ギャップ(数段階) / ホットキー有効化** 程度に絞る。閾値類は出さない。
2. **設定の永続化** — `UserDefaults` で上記トグル類を保存。現状ハードコード定数（`snap_zone.swift:11,13`, `window_dragger.swift:15`, `ax_window.swift:12`）は**デフォルト値として内部に残し**、ユーザー露出はしない（ギャップのみ設定可能化）。
3. **有効/無効トグル & ログイン時起動の GUI 設定** — `SMAppService`（macOS 13+）でログイン項目を管理し、現状の launchd plist 手動登録を置換（一般ユーザーに plist 運用は不可能）。
4. **主要スナップの最小ホットキー** — 左半分/右半分/最大化/1/4 等の**固定（または数プリセット）ホットキー**。Carbon `RegisterEventHotKey` 相当で登録。複雑な再割り当て UI は作らない。
5. **オンボーディング（初回起動フロー）** — アクセシビリティ権限付与をステップ UI で案内。非技術層が主対象のため**「ゼロ設定で即動く」体験は商品の核**であり P0。現状は 2秒ポーリングのみ（`main.swift:23-39`）。

### P1（体験の質・競争力 — 無料勢との差別化はここで作る）
6. **ドラッグ体験の作り込み** — プレビューの吸い付き・反応・誤爆のなさを磨く（`overlay_window.swift`/`snap_zone.swift`）。「安いだけでなく気持ちよく動く」が無料 Rectangle/Loop に勝つ核。
7. **1/3・2/3（thirds）と直前位置へ戻す（restore）** — カテゴリ標準として期待される範囲。`snap_zone.swift`/`geometry.swift` に追加。「マニアックなカスタム」ではなく標準スナップの拡充。
8. **マルチディスプレイ間のウィンドウ移動（ホットキー）** — 隣のディスプレイへ送る。既存基盤（`screen_util.swift`）を活用。
9. **README/ドキュメント整合と商品ページ素材** — `TITLEBAR_HEIGHT` 表記不一致（README:145=32pt／実コード44pt）、プレビュー「未実装」表記（README:154／実装済み）を修正。

### P2（余裕があれば — 方針上、優先度は低い）
10. **除外アプリ設定** — 特定アプリでスナップ無効化（バンドルID）。必要性が出たら。
11. **少数のローカライズ** — グローバル向けに主要言語のみ（観点2-P2 と連動）。

> 旧プランの「カスタムグリッド／レイアウト保存／モニタープロファイル」は**スコープ外**（Context 参照）。これらを足さないことが本商品の差別化。

## 観点2: 有料配布するために必要な機能

商品を「売って・届けて・課金して・保守する」ための土台。プロダクト機能とは別軸。

### P0（必須 — これが無いと販売・配布が成立しない）
1. **Apple Developer Program 登録 + Developer ID 署名** — 現状の自己署名（`scripts/create-signing-cert.sh`）では一般ユーザーの Mac で Gatekeeper に弾かれる。$99/年の登録と Developer ID Application 証明書での署名が必須。`scripts/build-app.sh` の署名 ID を差し替え。
2. **Notarization（公証）+ stapling** — Developer ID 署名済みアプリを Apple に公証申請（`notarytool`）し、チケットを `stapler` で添付。これが無いと初回起動でブロックされる。ビルドスクリプトに公証ステップを追加。
3. **配布パッケージ化（署名済み .dmg）** — `git clone`+ビルド前提を廃し、ダウンロードして Applications へドラッグできる .dmg を生成。launchd 手動登録も不要化（P0-3 の `SMAppService` と連動）。
4. **ライセンスキー発行・検証システム** — 買い切りなので**オフライン検証で十分**。発行側で秘密鍵署名したライセンス（メール/名前 + 公開鍵で検証）を生成し、アプリ内に公開鍵を埋め込んで検証。サーバ常時通信は不要。アプリ内に「ライセンスキー入力 UI」を追加。
5. **決済プラットフォーム連携** — **Merchant of Record（Paddle / Lemon Squeezy 等）を強く推奨**。グローバル販売では各国の VAT/消費税/インボイスを代行してくれるため。購入完了 Webhook → ライセンスキー自動発行・メール送信のフローを構築。Stripe 直結は税対応を自前で負う必要があり非推奨。
6. **自動アップデート機構（Sparkle）** — 直販アプリの事実上の標準。appcast XML を配信し、EdDSA 署名付きで安全更新。現状の「ソース変更検知で再ビルド」(`windowsnap-launch.sh`) は開発者専用なので置換。

### P1（販売の質・信頼性）
7. **試用（トライアル）機構** — **14日間フル機能トライアル → ライセンス購入**（フリーミアムの機能分割はしない＝シンプルさ訴求と整合）。初回起動日を保存し残日数を表示、期限切れでスナップを停止しライセンス入力を促す。改ざん耐性のため初回日時の保存場所に注意（時計巻き戻し対策は低価格帯なので軽量で可）。
8. **法務ドキュメント** — EULA / 利用規約 / プライバシーポリシー。グローバル販売・直販では必須。アクセシビリティ権限を使う旨の明示も含む。
9. **製品ランディングページ（LP）** — 機能紹介・スクリーンショット・購入ボタン・ダウンロード。決済プラットフォームと連結。
10. **サポート窓口 & ライセンス管理** — 問い合わせメール/FAQ、購入後のキー再送、デバイス数ポリシー。

### P2（運用の洗練）
11. **多言語ローカライズ** — グローバル向けに UI/LP を英語ベース＋主要言語へ。`Localizable.strings` 化。
12. **クラッシュレポート/匿名アナリティクス** — プライバシー配慮の上で導入（オプトイン）。改善判断材料。
13. **有料メジャーアップグレード機構** — v1→v2 で割引アップグレード価格を出せるライセンス世代管理（継続売上の柱）。

## 観点3: テスト容易性・テスト文化・CI（フェーズ0＝機能実装の前提として最初に行う）

機能実装に着手する前に、テストしやすい設計へ作り変え、テストを書く・必ず CI で回す・テスト漏れを検知する仕組みを整える。**UI はほぼ無いためスクリーンショットテストは行わない**。ロジック（純粋関数・状態機械）を XCTest で担保し、手動テストを最小化する方針。

### 現状の制約
- `Package.swift` は **executable ターゲット 1つのみ**（テストターゲット・ライブラリターゲットなし）。テスト対象を `@testable import` しづらい。
- `CLAUDE.md` 不在、`.github/workflows` 不在、`Tests/` 不在。
- ロジック（`snap_zone.swift`/`geometry.swift`）とシステム依存（`event_tap`/`ax_window`/`screen_util`/`overlay_window`/`status_item`/`main`）が同一ターゲットに同居。

### 0-1. テスト容易な設計へのリファクタ（最優先）
**ロジックを切り出し、システム依存を境界に追いやる**：
1. **`WindowSnapCore` ライブラリターゲットを新設**し、純粋ロジックを移設：
   - `snap_zone.swift`（ゾーン判定）, `geometry.swift`（座標変換・ゾーン矩形算出）を Core へ。これらはほぼ純粋関数で**最も価値が高くテストしやすい**。
   - スナップ閾値の定数をデフォルト値として Core に集約。
2. **システム依存をプロトコルで抽象化**し、状態機械の判断ロジックをテスト可能に：
   - `WindowControlling`（`ax_window.swift` の getFrame/setFrame を抽象化）→ テストではフェイク実装。
   - `ScreenProviding`（`screen_util.swift` の画面列挙/カーソル位置を抽象化）→ テストでは決定的なフレームを返すフェイク。
   - `window_dragger.swift` の状態機械（down→move→up でどのゾーンを選び・どの矩形に落とすか）を、上記プロトコル注入で純粋に判定できる形へ分離。
3. **executable ターゲット `WindowSnap` は薄いグルー層**に：`main`/`event_tap`/`overlay_window`/`status_item` と各プロトコルの実システム実装のみを残し、`WindowSnapCore` に依存させる。
4. `Package.swift` に **`WindowSnapCore`（library）/ `WindowSnap`（executable, Core依存）/ `WindowSnapCoreTests`（test）** の3ターゲット構成を定義。

### 0-2. テストコードの実装
- `Tests/WindowSnapCoreTests/` に XCTest を新設。**テーブルドリブン**で網羅：
  - ゾーン判定: 上端→最大化 / 左右→半分 / 四隅→1/4、`EDGE_THRESHOLD`/`CORNER_BAND` 境界値、下端中央が対象外であること、`MOVE_THRESHOLD` 未満はクリック扱い。
  - 座標変換: Cocoa↔CG の Y 反転、**マルチディスプレイ**（primary 高さ基準の global 変換）、`visibleFrame` ベースの menubar/Dock 回避。
  - 状態機械: down で対象確定後にドラッグ中も化けないこと、スペース切替で `cancelDrag`、サイズ制約ウィンドウのクランプ補正。
  - 機能追加分（thirds/restore、ギャップ、ホットキー→矩形マッピング、ライセンス検証、トライアル残日数判定）も**実装と同時にテストを追加**。
- `event_tap`/`overlay_window`/`status_item`/`main` 等の薄いシステムグルーは単体テスト対象外（手動テスト項目で担保）。

### 0-3. GitHub Actions でテストを必ず実行
- `.github/workflows/test.yml` — `macos-latest` 上で push / pull_request 時に `swift build` + `swift test --enable-code-coverage` を実行。**main への必須チェック（required status check）に設定**してマージをブロック。

### 0-4. テスト漏れ検知ワークフロー
- `.github/workflows/test-coverage-guard.yml`（または上記に統合）— PR で次を検査し、未達なら fail：
  - **変更検知**: `Sources/WindowSnapCore/` 配下が変更された PR で `Tests/` に変更/追加が無ければ fail（テストを書かずにロジックを変えた PR を弾く）。
  - **カバレッジ閾値**: `swift test --enable-code-coverage` の結果を `llvm-cov` で集計し、**`WindowSnapCore` のカバレッジが閾値（例: 80%）未満なら fail**。閾値・対象は Core 限定（システムグルーは除外）。
- 抜け道（テストファイルを空コミット等）は完全には塞げないため、**0-5 の CLAUDE.md ルールと併用**して機械＋規律の両輪で担保。

### 0-5. CLAUDE.md にルール追加（新規作成）
ルートに `CLAUDE.md` を新規作成し、以下を明文化：
1. **機能実装時は必ずテストコードも実装する。** ロジックは `WindowSnapCore` に置き XCTest で担保する。テスト不能なコードはシステム境界（executable 側のグルー）に限定する。
2. **PR 作成時は概要欄に必ず「自動テスト項目」「手動テスト項目」のテストケース一覧を記載する。** 自動テスト項目はテストコードで担保し（コードが無い自動テスト項目は不可）、**手動テスト項目は可能な限り減らす**（ロジックに落とせるものは Core へ移してテスト化する）。
3. **UI がほぼ無いためスクリーンショットテストは行わない。**
4. ロジック（テスト可能）とシステム依存（テスト対象外）を分離する設計を維持する。
5. CI（`swift test` + カバレッジ/テスト漏れ検知）が緑でないとマージしない。
6. **手動テストはフェーズ（サブフェーズ）単位で、そのフェーズの末尾に実施する。** 全フェーズ完了後にまとめて行わない。手動項目はそのフェーズの変更分に限定し最小化する。

### 0-6. PR テンプレート
- `.github/pull_request_template.md` を新設し、`## 自動テスト項目`（チェックリスト＋対応するテスト関数名）/ `## 手動テスト項目`（そのフェーズ分のみ・最小限）/ `## 手動を減らせなかった理由` セクションを雛形化。CLAUDE.md ルール2の運用を様式で強制。

### 0-7. 計画ドキュメントのリポジトリ保存
- 本ファイル（`docs/plans/pre-sales-roadmap.md`）がリポジトリの正本（canonical）。別セッション/別担当からも参照可能。以後の更新はこのファイルを直接編集する。

## 推奨ロードマップ（実装順）

**原則: 各フェーズは「小さく区切る・自動テストで担保する・末尾で最小限の手動テストを実施する」。** 手動テストは全フェーズ完了後にまとめてやらず、**そのフェーズの末尾でそのフェーズ分だけ実施**して累積負荷を下げる。ロジックに落とせるものは Core へ移して自動化し、各フェーズの手動項目を可能な限り少なく保つ。

> 手動テスト負荷とレビュー単位（≒PR単位）を小さくするため、当初の 0/A/B/C/D を**サブフェーズへ細分化**した。各サブフェーズ ≒ 1 PR を想定。「手動」は各サブフェーズ末尾に行う最小確認。

```
フェーズ0: テスト基盤（機能実装の前提・最初に必須）
  0a. ターゲット3分割リファクタ + snap_zone/geometry を Core へ + 既存ロジックのXCTest
      手動: 既存のドラッグスナップ回帰確認（左/右/最大化/四隅/マルチディスプレイ）一巡のみ
  0b. CI(test.yml) + テスト漏れ検知 + カバレッジ閾値 + main必須チェック設定
      手動: なし（CI挙動はわざと失敗PRを作って自動確認）
  0c. CLAUDE.md + PRテンプレート + 本計画を docs/plans/ へ保存
      手動: なし

フェーズA: 配布の土台（一般ユーザーが入れて使える状態に）
  A1. SMAppService 起動管理 + 有効/無効トグル + 最小設定UI骨格
      手動: ログイン項目の登録/解除、ON/OFFトグルの効き
  A2. オンボーディング（権限付与フロー）
      手動: 権限未付与のクリーン環境/別ユーザーでの初回起動フロー
  A3. Developer ID署名 + 公証 + .dmg
      手動: 別Mac/別ユーザーで Gatekeeper 警告なし起動（spctl/stapler）

フェーズB: 課金の土台（B完了＝販売開始の最小ライン＝MVP）
  B1. オフラインライセンス検証 + キー入力UI（検証ロジックはCoreでテスト）
      手動: 正規/無効/改ざんキーの判定確認
  B2. 14日トライアル（残日数判定ロジックはCoreでテスト）
      手動: 残日数表示・期限切れ挙動
  B3. 決済連携(MoR) + Webhook→キー自動発行・送信
      手動: テストモードでのE2E購入→受信
  B4. Sparkle自動更新
      手動: 旧→新の更新を1回
  B5. EULA/プライバシーポリシー + LP
      手動: なし（文面レビューのみ）

フェーズC: 無料勢に勝つための体験の質
  C1. thirds/restore（ロジックはCoreでテスト）
      手動: 1/3・2/3・restore の動作確認
  C2. 主要スナップの最小ホットキー（矩形マッピングはCoreでテスト）
      手動: 各ホットキー発火確認
  C3. ドラッグ体験の作り込み（吸い付き/プレビュー/誤爆のなさ）
      手動: 体感確認（ここは手動比率が上がるため小さく区切る）
  C4. ディスプレイ間移動ホットキー
      手動: マルチディスプレイでの移動
  C5. ドキュメント整合（README/商品ページ素材）
      手動: なし

フェーズD: 余裕があれば
  D1. 除外アプリ設定 / D2. 少数ローカライズ / D3. 匿名アナリティクス（オプトイン）
```

ポイント: **フェーズ0 を最初に完了**（テスト土台が無いまま積むと作り直しになる）。**フェーズB 完了が販売開始の最小ライン（MVP）**。**フェーズC の "ドラッグ体験の質"（特に C3）がこの商品の本当の差別化**であり、安さ・シンプルさは入口に過ぎない。手動比率が高い C3 を独立サブフェーズにして、それ以外を自動テスト主体に保つのが細分化の狙い。

## 検証方法（フェーズ単位で実施）

> 各サブフェーズの末尾で、そのフェーズ分の自動テスト（CI 緑）＋下記の最小限の手動テストを実施する。手動項目はロードマップの各サブフェーズ「手動:」行と対応。以下は代表的な確認観点。

0. **テスト基盤（フェーズ0）**: ローカルで `swift build` と `swift test` が通る。`WindowSnapCore` のカバレッジが閾値以上。PR で「Sources/WindowSnapCore を変えたのに Tests 未変更」を意図的に作ると CI が fail する。PR テンプレートに自動/手動テスト項目欄が表示される。main の必須チェックに `test` が登録され、赤だとマージできない。
1. **配布パッケージ（A3）**: 別の（または初期化した）Mac、最低でも別ユーザーアカウントで .dmg をダウンロード→インストール→**Gatekeeper の警告なしに起動できる**ことを確認（`spctl -a -vvv WindowSnap.app` で `accepted` / `source=Notarized Developer ID`）。
2. **署名・公証（A3）**: `codesign --verify --deep --strict` と `xcrun stapler validate WindowSnap.app` が通る。
3. **設定 UI / 永続化（A1・C1・C2）**: 設定値を変更→再起動→保持されること。ホットキー・ドラッグ両方でスナップ動作（左/右/最大化/thirds/1/4/restore）を実機確認。マルチディスプレイでも確認。
4. **ライセンス（B1・B2）**: 正規キーで全機能解放、無効/改ざんキーで拒否、トライアル期限切れ挙動を確認（オフラインでも検証が成立すること）。
5. **決済→発行フロー（B3）**: 決済プラットフォームのテストモードで購入→Webhook→キー自動メール送信までを E2E で確認。
6. **自動更新（B4）**: 旧バージョンから appcast 経由で新バージョンへ Sparkle 更新が成功すること（EdDSA 署名検証含む）。
